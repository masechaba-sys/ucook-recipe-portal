UPDATED — Kate's feedback
Test & Shoot Flags
Always Serves 2 (S2)
When marked for shoot, the procurement team will order S2 quantities only. Quantities and SKUs are fixed to the Serves 2 portion.
Select serves to test
Select which serve sizes will be tested. Multiple combinations allowed (e.g. S1 only, S1+S3, S2 only).
Available at menu approval
Tab bar — 5 tabs now Added "Ingredients" and "Method" as top-level tabs. This de-clutters the Recipe tab which was previously overwhelming.
Recipe tab — simplified Recipe tab now only contains: Planner assignment, Cost overview cards, Dish name/category/protein, Description, Tags, and Test/Shoot flags.
Cost cards — keep The 3 cost summary cards (JHB, CPT, Avg) stay on the Recipe tab as a persistent summary — not moved to Costing screen.
📸 Shoot flag — always S2 When a dish is flagged for shoot, it always uses Serves 2 (S2) quantities. The "Generate Shoot Order List" button creates a procurement order in the same format as existing order lists.
🧪 Test flag — flexible serves Chefs select which serves to test — any combination of S1, S2, S3, S4 is valid. At menu approval, "Generate Testing Sheet" and "Generate Portioning Sheet" become active.
Portioning Sheet The Portioning Sheet (used for mock bagging) is derived from the same Testing Sheet / Testing Order List — not a separate data entry.

⚠️
S1 vs S2 as the base SKU for Protein Mapping

Currently the food team maps on S1. Kate proposes switching to S2 so it flows through to Recipe Writing and Shoot Orders automatically.

Option A — S2 (proposed)Consistent with shoot quantities. Clean end-to-end flow.
Option B — S1 (current)Depends on Cornell's costing source — confirm before changing.
Spreadsheet-style layout The protein map replicates the existing Google Sheet used by the food team: category groups (FOODIE, Q&E, FF, CC) as row spans, weeks as column groups, each week split into Protein ▾ / CW / Ave.p/p sub-columns.
Category rows & pricing Each category (FOODIE, Q&E, FF, CC) spans multiple meal-plan sub-rows and shows the target dish cost in the second column, matching the existing planning sheet.
[*] Special / procured items Cells highlighted in salmon with a [*] prefix denote externally procured or special proteins (e.g. Fish Swordfish, FF Burger Patty, Beef Wagyu Mince). These map to the pink-highlighted cells in the current spreadsheet.
CW & Ave.p/p sub-columns Each week column shows Cook Week frequency (CW) and average cost per person (Ave.p/p at S2) alongside the protein dropdown, preserving the data density of the source spreadsheet.
⚠️ Open: S1 vs S2 base SKU Kate's proposal: switch protein mapping from S1 to S2. Needs confirmation from Cornell re: costing data source before implementing.

⏳ Testing Sheet and Portioning Sheet unlock once the menu is approved at the end of costing. Shows quantities for each selected serve (S1, S3 in this example).
Shoot Order List Generated immediately when a dish is marked for shoot. Always uses S2 (Serves 2) quantities and SKUs. Same table format as existing proc team order lists.
Testing Sheet Shows quantities for each selected test serve (e.g. S1 and S3). Generated by the prop team at menu approval. Columns appear only for the selected serves — if only S1 was chosen, only one qty column appears.
Portioning Sheet Used for mock bagging. Derived from the same Testing Sheet data — not a separate data source. Both documents can be generated from the same "Generate Testing Sheet" action.
Timing / availability Shoot Order List → available immediately.
Testing Sheet + Portioning Sheet → locked until menu approval in Costing.
Format consistency All three documents should match the existing order list format the proc team already uses. No new format to learn.
